## 二、做得好的

**1. 快速从想法到上线**
从"帮我分析一下素材"到"产品上线公开可访问"，2 天内完成。核心原因是保持技术栈极简——单文件 HTML，不引入框架，不做后端。

**2. 调研驱动的产品设计**
初期用 Web 搜索验证了外贸业务员的痛点（自称"打字机器"）、竞品缺口（OKKI $50-200/月 vs 个人买断 199 元）、付费信号（阿里 AI 夜间询盘回复率提升 39%）。没有凭空想象。

**3. 数据看板 + 跟进看板**
这两个功能让 TradeMate 从"写邮件工具"升级为"邮件工作管理平台"。数据看板让用户看到自己的使用轨迹，跟进看板解决了"发完就忘"的痛点。

---

## 三、做得不好的

**1. 语言问题反复了 8 次**
这是整个项目最严重的效率黑洞。根本原因：一开始没有想清楚"系统语言"和"输出语言"的区别，每次只修表面 bug 不解决根源。直到用户明确指出三者的关系（系统语言=UI、客户语言=输入、回复语言=输出），才彻底解决。

**2. 设计迭代过多**
G2 设计阶段来回改了 5 版——侧边栏/无侧栏/双列/单列，每次都是修修补补而不是彻底重想。根源：一开始就在改细节（颜色、间距），没有先确定布局方向。

**3. 测试不够系统**
多个 bug 是在用户使用中发现的（变量名冲突 D.ts、语言不生效、历史记录 HTML 注入）。应该在开发完成后做一次完整的功能走查，而不是依赖用户反馈。

**4. "临时方案"变成"永久方案"**
语言翻译最初只想支持中英，后来不断增加，但数据结构没有相应调整。最终 T 对象膨胀到 22 个语言 × 35 个键，维护成本很高。

## 四、经验教训

| 教训 | 具体行动 |
|------|---------|
| 核心概念先对齐再开发 | 遇到语言问题先停下来画关系图，而不是改代码 |
| 设计先定布局再定细节 | 侧边栏 vs 无侧栏这种决定必须在调颜色之前做 |
| 每个阶段结束时做完整走查 | 不是"看起来能跑就行"，要逐项验证 |
| 保持技术栈极简 | 单文件 HTML 让我们能在 2 天内完成全部迭代 |
| 用户反馈是最好的测试 | 多个 bug 是用户发现的，说明自己测试不充分 |
